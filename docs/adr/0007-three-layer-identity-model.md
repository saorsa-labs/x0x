# ADR 0007: Three-Layer Identity Model

- Status: Accepted
- Date: 2026-04-24

## Context

x0x exposes several identifiers that are easy to confuse if they are all described as "the identity":

- `machine_id`
- `agent_id`
- `user_id`

They are intentionally different. The transport needs a cryptographic identity for authenticated QUIC connections and NAT traversal. The application needs a stable logical identity that can move between machines. Some deployments also need an optional human identity, but x0x must not create or disclose that identity automatically.

The system also has multiple identity-bearing artifacts:

- local key files;
- identity announcements;
- agent certificates;
- contact machine records;
- direct-message and gossip-DM envelopes;
- shareable agent cards.

Without a clear architectural decision, documentation and product surfaces can overstate what a single ID proves.

## Decision

x0x SHALL use a three-layer identity model:

1. `MachineId` is the transport identity.
   - It is `SHA-256(machine ML-DSA-65 public key bytes)`.
   - It is derived from `machine.key`.
   - It is the same value as the authenticated ant-quic `PeerId`.
   - It authenticates the remote machine on QUIC connections.

2. `AgentId` is the logical agent identity.
   - It is `SHA-256(agent ML-DSA-65 public key bytes)`.
   - It is derived from `agent.key`.
   - It is portable across machines when the agent key is moved.
   - It is the main long-lived identifier users and agents exchange.

3. `UserId` is an optional human identity.
   - It is `SHA-256(user ML-DSA-65 public key bytes)`.
   - It is derived from `user.key`.
   - It is never generated by default.
   - It is disclosed only when a user key is configured and the caller gives explicit human consent.

When a user key exists, x0x issues an `AgentCertificate` signed by the user key over the user public key, agent public key, domain-separation context, and issuance timestamp. The certificate binds a human user key to a specific agent key. It does not replace the machine key or the agent key.

Identity announcements SHALL bind the portable `AgentId` to the current `MachineId` by signing the announcement with the machine key and including the machine public key. If a `UserId` is included, the announcement MUST also include a valid `AgentCertificate` whose derived user and agent IDs match the announcement.

Trust and routing decisions SHALL evaluate the `(AgentId, MachineId)` pair rather than either ID alone. Machine pinning constrains a known `AgentId` to one or more accepted `MachineId`s.

## Operational Rules

Default unnamed instances store identity material under `~/.x0x`:

```text
machine.key
agent.key
user.key      # optional
agent.cert    # optional
```

Named daemon instances use separate identity directories such as `~/.x0x-alice` and `~/.x0x-bob`.

`machine.key` and `agent.key` are generated when missing. `user.key` is not generated automatically. Create it explicitly with `x0x user-id create [PATH]`; without `PATH`, the command writes `~/.x0x/user.key`, overwriting any existing target file.

If an existing `agent.cert` does not match the configured user key and current agent key, x0x should treat it as stale and issue a new certificate when a user key is active. Without an active user key, the certificate is inert local state.

Shareable agent cards are identity metadata and contact-import aids. They are not key backups and are not, by themselves, proof of a human-backed identity.

## Consequences

### Positive

- Transport authentication and logical agent identity are not conflated.
- A logical agent can move hardware without changing `agent_id`.
- Machine pinning has a precise meaning: this agent is expected on this machine.
- Human identity remains explicitly opt-in.
- Discovery and direct-message verification can reason about both the authenticated transport machine and the claimed logical agent.

### Negative

- Users may see more than one ID for the same daemon.
- Moving an agent safely requires understanding which key files are being moved.
- A stale optional `agent.cert` can exist on disk after key changes, although it is revalidated before use with an active user key.
- Agent cards are convenient but must not be treated as complete identity proof.

## Non-goals

- This ADR does not define key recovery.
- This ADR does not define key rotation while preserving the same ID.
- This ADR does not create a global account system.
- This ADR does not require user identity for ordinary agent operation.
- This ADR does not make `AgentCard` a signed introduction-card replacement.

## Acceptance Criteria

This ADR is satisfied only when:

- documentation distinguishes `machine_id`, `agent_id`, and `user_id`;
- user identity is documented as opt-in and consent-gated;
- docs explain that `machine_id` equals the ant-quic transport `PeerId`;
- docs explain that `agent_id` is the portable day-to-day identity;
- docs explain that trust/machine pinning evaluates `(AgentId, MachineId)`;
- docs do not present agent cards as key backups or standalone human identity proof.
